# Keychain 10 升级与生物识别行为说明

## 背景

本次 PR 的目标是引入 `react-native-keychain@10.0.0` 的可选实现，同时保留当前线上用户可感知的生物识别行为。更严格的“系统生物凭据变化后让 App 生物解锁失效”不在本次 PR 范围内，需要产品确认后单独设计。

## 升级前

- App 业务层已有 `8.2.0-fork` 和 `9.0.0` 两套 keychain 实现。
- 当前默认实现是 `9.0.0`，开发/测试环境可以通过调试选项切换版本。
- 业务层的普通生物识别类型是 `KEYCHAIN_AUTH_TYPES.BIOMETRICS`，写入 keychain 时传 `BIOMETRY_CURRENT_SET`。
- Android 线上行为并不是“绑定开启 Rabby 时那一组指纹”。只要系统仍有可用强生物识别，新增后的指纹也可以通过系统认证，并继续取出 App 存储的密码。
- iOS 使用系统 Keychain 的 current set 语义。Face ID 设备通常没有新增第二套生物凭据的高频场景，所以这个差异在线上不明显。

## 本次 PR 后

- 新增 `10.0.0` keychain 业务封装，业务层可以在 `8.2.0-fork`、`9.0.0`、`10.0.0` 间选择实现。
- 默认版本仍保持 `9.0.0`，本次 PR 不直接把全量用户切到 10。
- `react-native-keychain@10.0.0` 使用 Yarn patch 引入 Rabby 需要的 Android 调试、存储兼容和行为修复。
- `9.0.0` 被保留为独立 workspace package，避免与官方 10 的原生模块、iOS 符号和调试入口冲突。
- Android 10 路径支持 RSA、AES-CBC、AES-GCM 等存储调试/验证能力，但不主动加入“生物凭据集合变化检测”。
- 生物识别产品语义继续对齐当前线上方案：用户通过系统认可的生物识别后，可以解锁 App 并取出已保存密码。

## 本次 PR 明确不做的事

- 不检测用户是否新增了指纹、面容或其他生物凭据。
- 不在 Android 上新增 enrollment sentinel key。
- 不在新增/删除生物凭据后自动关闭 Rabby 生物识别。
- 不要求用户因为系统生物凭据变化而重新输入 App 密码。
- 不改变当前自动生成密码用户的兜底策略。

## 后续产品建议

后续如果希望提升安全性，可以单独讨论“生物凭据变化后重新授权”的产品策略。这个策略会改变用户体验，需要先明确平台差异和恢复路径。

建议讨论点：

1. 是否要求 Android 和 iOS 使用同一安全语义：系统生物凭据变化后，Rabby 生物解锁失效。
2. 用户新增指纹后是否必须输入 App 密码重新开启生物识别。
3. 自动生成密码用户在生物识别失效后的恢复路径是否足够清晰。
4. 是否需要在设置页说明“系统生物识别凭据变化后需要重新授权”。
5. 是否需要埋点区分：无生物凭据、用户取消、系统凭据变化、KeyStore/Keychain 数据不可读。

如果采用更严格方案，技术上可以这样做：

- Android：创建一把仅用于检测的 Keystore sentinel key，要求每次使用都走强生物识别，并让它在 biometric enrollment 变化后失效。App 启动、回前台或解锁前尝试使用该 key，若发现 key 失效，则关闭 Rabby 生物解锁并要求用户用 App 密码重新开启。
- iOS：结合 `LAContext.evaluatedPolicyDomainState` 或 current set Keychain 失败结果判断生物凭据集合变化。
- 两端都需要补齐 UI 文案、失败恢复和 QA 用例，避免用户误以为资产或密码丢失。

## QA 建议

- Android：开启生物识别后新增一个指纹，确认新增指纹仍可解锁，这是本次 PR 期望保留的线上行为。
- Android：删除全部生物凭据后，确认 App 不再展示可直接使用的生物识别能力，并能回落到 App 密码路径。
- iOS：验证 Face ID / Touch ID 开启、解锁、关闭流程不回归。
- 版本切换：分别验证 `9.0.0` 与 `10.0.0` 下开启、验证、关闭生物识别的基础流程。
